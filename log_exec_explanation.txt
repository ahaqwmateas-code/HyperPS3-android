Technical Report: APS3E Execution Failure Analysis (DMC4)
Target Hardware: MediaTek Dimensity 6100+ (GPU: Mali-G57 MC2)
OS: Android 14 (Samsung A15 variant)

---

1. Primary Cause of Crash: GPU Format Mismatch
The most critical error identified in the logs originates from the mali_gralloc driver and the emulator's native engine (libe.so).

* Error Signature: E/mali_gralloc: ERROR: Unrecognized and/or unsupported format 0x38 and usage 0xb00.
* Analysis: The emulator is attempting to allocate a memory buffer using format 0x38 (Decimal 56). In the Vulkan specification, this often corresponds to VK_FORMAT_R8G8B8A8_UNORM.
* Conflict: The Mali-G57 (Valhall architecture) driver is rejecting this specific format when combined with the memory usage flags 0xb00. This usually happens when the emulator tries to create a surface for the "Blit" operation (rendering the PS3 frame to the Android screen) using a format that the mobile GPU's hardware composer cannot handle.

2. Native Library Linkage Issues
A secondary, yet significant, error was detected during the library loading phase:
* Error: E FBI: Can't load library: dlopen failed: library "libmagtsync.so" not found.
* Analysis: libmagtsync.so is a MediaTek-proprietary library used for hardware buffer synchronization. The emulator (or one of its 3rd party dependencies like FFmpeg or a custom Vulkan layer) is hardcoded to look for this library. Its absence likely causes a fallback to a null pointer, eventually leading to a crash when the engine attempts to sync a frame.

3. Vulkan Implementation Concerns (Mali Specific)
* Texel Buffer Limits: In VKGSRender.cpp, the logic for m_texbuffer_view_size was modified to use limits.maxTexelBufferElements. While this is a good optimization for Mali, the code contains a fallback that forces a large size if a specific config is disabled. If this forced size exceeds the hardware limits of the G57 (which only has 2 execution cores), the driver will terminate the Vulkan instance (VK_ERROR_DEVICE_LOST).
* Unused Format Parameters: In buffer_object.cpp, the sub_buffer constructor receives a VkFormat but never applies it to the VkBufferCreateInfo. Mali GPUs are highly dependent on explicit format declarations for memory alignment. Passing "raw" buffers without properly defined formats for sub-resources can lead to the "Invalid base format" errors seen in the logs.

4. Project Environment Observations
* AdrenoTools Integration: The project currently includes libadrenotools. This library is designed for Qualcomm Adreno GPUs to allow custom driver loading. On a MediaTek/Mali device, this code is at best redundant and at worst causing initialization conflicts during the vk_load phase in vkapi.cpp.

---

Recommendations for Improved Mali Support:
1. Format Fallback: Implement a check in the Vulkan swapchain initialization to force VK_FORMAT_R8G8B8A8_UNORM if the driver rejects more exotic PS3-native formats.
2. Strict Limit Validation: Ensure m_texbuffer_view_size never exceeds limits.maxTexelBufferElements regardless of the configuration flags.
3. Mali-Specific Sync: Investigate if the dependency on libmagtsync.so can be replaced with standard Vulkan Fences or Semaphores to ensure compatibility across different MediaTek driver versions.
